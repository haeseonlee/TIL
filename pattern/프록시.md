# 프록시(Proxy)

## 프록시(Proxy)란?
프록시 패턴(Proxy Pattern)은 GoF 디자인 패턴 중 하나로, 클라이언트와 타겟(실제 객체 또는 서버) 사이에 요청을 대신 받아 전달하는 중간 대리자 객체를 의미한다.

## 프록시를 사용하는 주요 목적
대리자를 중간에 두면 타겟 객체의 코드를 직접 수정하지 않고도 다음과 같은 핵심적인 부가 기능을 추가할 수 있다.

- **접근 제어 및 권한 검사**: 특정 조건이나 권한을 만족하는 요청만 실제 객체로 전달한다.
- **캐싱**: 이전에 이미 처리했던 요청이라면 실제 객체까지 가지 않고 프록시가 저장해둔 데이터를 즉시 반환하여 성능을 최적화한다.
- **부가 기능 추가(로깅, 트랜잭션 등)**: 실제 객체가 실행되기 전후로 로그를 남기거나, 데이터베이스 트랜잭션을 시작하고 종료하는 등의 공통 로직을 삽입할 수 있다. 
- **지연 로딩(Lazy Loading)**: 객체 생성 비용이 아주 큰 경우, 실제 객체가 정말로 사용되는 시점까지 생성을 미루고 프록시가 임시로 자리를 대체한다.

## 디자인 패턴에서의 프록시
GoF 디자인 패턴에서는 프록시를 사용하는 의도에 따라 크게 프록시 패턴과 데코레이터 패턴으로 구분한다. 두 패턴은 구현 구조(인터페이스를 두고 대리 객체를 합성하는 방식)가 거의 동일하지만, 사용 목적에서 명확한 차이가 있다. 

1. 프록시 패턴(Proxy Pattern)
- 목적: 실제 객체에 대한 접근 제어가 주요 목적이다.
- 특징: 프록시가 실제 객체의 인스턴스를 제어하며, 필요에 따라 객체를 생성하거나 접근을 차단할 수 있다. 클라이언트는 실제 객체의 반환 값을 변경 없이 그대로 받는다.

2. 데코레이터 패턴(Decorator Pattern)
- 목적: 실제 객체가 가진 기능에 새로운 행동(기능)을 동적으로 추가하는 것이 주요 목적이다.
- 특징: 프록시 패턴과 달리 접근을 제어하기보다는, 실제 객체의 반환값에 장식으로 더하거나 새로운 로직을 결합하여 기능을 확장한다.

## Spring에서의 프록시
Spring AOP는 프록시를 사용하여 AOP 기능을 구현한다. 프록시 객체는 타켓 객체(Target Object)를 감싸고 있으며, 메서드 호출을 가로채 부가 기능을 적용한다.

## 프록시의 종류

### JDK 동적 프록시(JDK Dynamic Proxy)
Java 프레임워크 자체에서 제공하는 기능으로, 개발자가 직접 프록시 클래스를 만들지 않아도 런타임에 동적으로 프록시를 생성해준다.

#### 특징 및 동작방식
- **인터페이스 필수**: JDK 동적 프록시는 반드시 인터페이스를 기반으로 프록시를 생성한다. 인터페이스가 없다면 이 방식을 사용할 수 없다.
- **InvocationHandler 사용**: 프록시 객체의 메서드가 호출되면, 내부적으로 InvocationHandler 인터페이스의 invoke() 메서드가 실행되면서 부가기능(공통 로직)과 타겟 객체의 핵심 로직이 수행된다.
- **프록시의 위치**: 생성된 프록시 객체는 타겟 객체와 동등한 위치에서 동일한 인터페이스를 구현한다.

```
// 1. 인터페이스
public interface Service {
    void execute();
}

// 2. 타겟 클래스
public class RealService implements Service {
    @Override
    public void execute() {
        System.out.println("핵심 로직 실행");
    }
}

// 3. 사용 예시
public class JdkProxyTest {
    public static void main(String[] args) {
        Service target = new RealService();
        
        Service proxy = (Service) java.lang.reflect.Proxy.newProxyInstance(
            Service.class.getClassLoader(),
            new Class[]{Service.class},
            (proxyObj, method, args1) -> {
                System.out.println("--- 로깅 시작 ---");
                Object result = method.invoke(target, args1); // 타겟 호출
                System.out.println("--- 로깅 종료 ---");
                return result;
            }
        );
        
        proxy.execute();
    }
}
```

#### 장점
별도의 외부 라이브러리 의존성이 필요 없으며, 인터페이스를 통한 느슨한 결합을 유지할 수 있다.

#### 단점
인터페이스를 적용하지 않은 일반 클래스(구체 클래스)에는 적용이 불가능하다.

### CGLIB 프록시(Code Generation Library)
CGLIB은 클래스의 바이트코드를 조작하여 동적으로 서브 클래스(자식 클래스)를 생성해주는 오픈소스 라이브러리다. 스프링은 이를 내장하여 구체 클래스만 있는 경우에도 프록시를 생성할 수 있도록 지원한다.

#### 특징 및 동작방식
- **클래스 기반 바이트코드 조작**: 인터페이스가 없어도 구체 클래스를 직접 상속받아 프록시 객체를 생성한다.
- **MethodInterceptor 사용**: 프록시 객체의 메서드가 호출되면 MethodInterceptor 인터페이스의 intercept() 메서드가 실행되어 부가 기능을 처리한다.
- **프록시의 위치**: 생성된 프록시 객체는 타겟 클래스의 자식 클래스(Subclass)가 된다. 

```
import org.springframework.cglib.proxy.Enhancer;
import org.springframework.cglib.proxy.MethodInterceptor;

// 1. 타겟 클래스 (인터페이스 없음)
public class ConcreteService {
    public void execute() {
        System.out.println("핵심 로직 실행");
    }
}

// 2. 사용 예시
public class CglibTest {
    public static void main(String[] args) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(ConcreteService.class); // 부모 클래스 지정
        enhancer.setCallback((MethodInterceptor) (obj, method, args1, proxy) -> {
            System.out.println("--- 로깅 시작 ---");
            Object result = proxy.invokeSuper(obj, args1); // 부모 클래스의 메서드 호출
            System.out.println("--- 로깅 종료 ---");
            return result;
        });

        ConcreteService proxy = (ConcreteService) enhancer.create();
        proxy.execute();
    }
}
```

#### 장점
인터페이스가 없는 구체 클래스에도 프록시를 적용할 수 있으며, 메서드 호출 시 리플렉션을 사용하는 JDK 동적 프록시보다 상대적으로 성능이 미세하게 우수하다. 

#### 단점
상속을 이용하기 때문에 타켓 클래스가 final이면 프록시를 생성할 수 없다. 상속 구조상 부모 클래스의 기본 생성자가 필요하다.
