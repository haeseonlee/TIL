# AOP와 트랜잭션 동작원리

## AOP란?
- AOP(Aspect-Oriented Programming)이란 애플리케이션 전체에 걸쳐 반복되는 공통 관심사(Cross-cutting Concerns)를 비즈니스 로직(Core Concerns)와 분리하여 모듈화하는 프로그래밍 패러다임이다.
  - 핵심 관심사: 회원 가입, 상품 주문, 결제 등 비즈니스 고유의 로직
  - 공통 관심사: 로깅, 보안 인증, 트랜잭션 관리(반복적인 커밋/롤백 처리)
- 객체 지향 프로그래밍에서는 주요 관심사에 따라 클래스를 분할한다. 하지만 클래스를 설계하다보면 로깅, 보안, 트랜잭션 등 여러 공통 관심사가 생긴다. 이는 주요 비스니스 로직은 아니지만, 반복적으로 여러 곳에서 쓰이는데 이를 흩어진 관심사라고 한다.

## AOP 없이 흩어진 관심사를 처리하면?
- 비즈니스 코드가 복잡해지고, 중복 코드가 수없이 발생하며 유지보수가 힘들어진다.
- 주요 비즈니스 로직이 부가 기능이 한 곳이 섞여 가독성이 떨어진다.

```
public void registerMember(Member member) {
    EntityTransaction tx = em.getTransaction();
    try {
        tx.begin(); // 공통 관심사 (트랜잭션 시작)
        
        memberRepository.save(member); // 핵심 비즈니스 로직
        
        tx.commit(); // 공통 관심사 (커밋)
    } catch (Exception e) {
        tx.rollback(); // 공통 관심사 (롤백)
        throw e;
    }
}
```
- AOP를 사용하면 이 공통적인 트랜잭션 처리 로직을 한 곳에 묶어두고, 비즈니스 로직은 순수하게 핵심 기능에만 집중할 수 있게 된다.

## Spring AOP는 왜 프록시(Proxy)를 사용하는가?
스프링은 대상 객체(Target)의 코드를 직접 수정하지 않고 AOP를 구현하기 위해 프록시 패턴을 사용한다.

### 프록시(Proxy)를 사용하는 이유
- 자바 언어 자체를 뜯어고치거나, 매번 컴파일 시점에 특수한 처리를 하는 것은 매우 복잡하다. 반면, 스프링은 런타임 시점에 프록시를 동적으로 생성하여 이 문제를 해결한다. 
  - **접근 제어 및 기능 추가**: 클라이언트가 직접 객체(Target)을 호출하는 것이 아니라, 겉모습이 똑같이 생긴 가짜 객체(Proxy)를 호출하게 만든다.
  - **호출 가로채기(Intercept)**: 프록시는 클라이언트의 요청을 가로채서 본인 내부에 있는 공통 로직(트랜잭션 시작 등)을 먼저 시작한 뒤, 실제 객체의 비즈니스 메서드를 호출한다.  

**참고(스프링의 프록시 생성방식 2가지)**
  - JDK Dynamic Proxy: 타겟이 인터페이스를 구현하고 있을 때 인터페이스 기반으로 프록시 생성
  - CGLIB: 인터페이스가 없을 때(또는 스프링부트 기본 값). 타켓 클래스를 상속받아 바이트코드를 조작하여 프록시 생성
  
## @Transactional은 어떻게 동작하는가?
스프링은 `@Transactional` 어노테이션을 발견하면 내부적으로 트랜잭션 AOP 프록시를 적용한다.
```
@Service
public class MemberService {

    @Transactional
    public void join(Member member) {
        // 핵심 비즈니스 로직만 작성!
        memberRepository.save(member);
    }
}
```

## 핵심 컴포넌트 4가지
- **트랜잭션 프록시(Proxy)**: 클라이언트의 요청을 가장 먼저 받아 트랜잭션 처리를 대행하는 객체이다. 스프링은 기본적으로 CGLIB이나 JDK Dynamic Proxy를 사용해 타켓 클래스를 감싸는 프록시를 생성한다.
- **트랜잭션 인터셉터(TransactionInterceptor)**: 프록시 내부에서 실질적인 트랜잭션 로직(시작,커밋,롤백)을 호출하는 인터셉터이다. 
- **트랜잭션 매니저(PlatformTransactionManager)**: 구체적인 트랜잭션 기술(JDBC, JPA 등)에 의존하지 않고 트랜잭션을 관리할 수 있도록 추상화된 기술이다.
- **트랜잭션 동기화 매니저(TransactionSynchronizationManager)**: 트랜잭션이 시작되면 획득한 데이터베이스 커넥션을 스레드 로컬(ThreadLocal)에 보관하여, 같은 스레드 내에서 실행되는 비즈니스 로직들이 동일한 비즈니스 커넥션을 공유할 수 있도록 돕는다.

## 내부 동작 흐름
클라이언트가 `@Transactional`이 적용된 메서드를 호출하면 다음과 같은 순서로 내부 매커니즘이 작동한다.

#### 1. 프록시(Proxy) 호출
- 클라이언트가 타겟 객체의 메서드를 호출하면, 실제 타켓이 아닌 프록시 객체의 메서드가 대신 호출된다. 

#### 2. 트랜잭션 시작 요청
- 프록시 내부의 트랜잭션 인터셉터(TransactionInterceptor)가 작동하여 트랜잭션 매니저(PlatformTransactionManager)에게 트랜잭션을 시작해달라고 요청한다.

#### 3. 커넥션 확보 및 동기화
- 트랜잭션 매니저는 데이터 소스(DataSource)로부터 DB 커넥션을 가져온다.
  - 수동 커밋 모드(setAutoCommit(false))로 변경하여 트랜잭션을 물리적으로 시작한다.
  - 이 커넥션을 트랜잭션 동기화 매니저(TransactionSynchronizationManager)에 보관한다.

#### 4. 비즈니스 로직(타켓) 수행
- 프록시는 실제 타겟 객체의 메서드를 호출한다.
  - 이떄 비즈니스 로직 내부에서 실행되는 Repository나 DAO 등은 트랜잭션 동기화 매니에서 보관중인 커넥션을 꺼내어 사용함으로써 하나의 트랜잭션으로 묶이게 된다.

#### 5. 결과에 따른 처리
- 정상 종료: 비즈니스 로직이 예외 없이 끝나면 트랜잭션 매니저가 DB 커넥션을 통해 Commit을 수행한다. 
- 예외 발생: 로직 수행 중 RuntimeException 등이 발생하면 Rollback을 수행한다. 

## 프록시 방식의 주의점: 내부 호출(Self-Invocation) 문제
- 프록시 방식의 특성 때문에 개발할 때 주의해야하는 사항이 있다.

#### 내부 호출 시 트랜잭션 미적용
- 하나의 클래스 내부에서 `@Transactional`이 없는 메서드가, 동일한 클래스 내부의 `@Transactional`이 붙은 메서드를 호출하면 트랜잭션이 적용되지 않는다.
  - 원인: 스프링의 트랜잭션의 클라이언트가 외부에서 프록시 객체를 호출할 떄만 가로채기(Intercept)가 가능하다. 클래스 내부에서의 메서드 호출은 프록시를 거치지 않고 `this.method()` 형태로 타겟 객체 자체의 메서드를 직접 호출하기 때문에 트랜잭션 인터셉터가 동작할 기회가 없다.
