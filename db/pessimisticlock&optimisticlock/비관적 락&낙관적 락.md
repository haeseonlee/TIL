# 비관적 락 & 낙관적 락

## 동시성 문제(Concurrency)란?
- 멀티스레드 환경에서 여러 트랜잭션이 동일한 자원(데이터베이스 등)에 동시에 접근하여 수정(write)하려고 할 떄, 데이터의 일관성과 정합성이 깨지는 현상이다. 
- 트랜잭션이 데이터를 읽거나 수정할 때, 다른 트랜잭션이 동일한 데이터베이스에 접근하지 못하도록 하는 것이 중요하다. 

## DB 락(Lock)이란?
- 데이터베이스의 동시성을 제어하고 데이터 무결성을 보장하기 위해 사용된다.
- 여러 커넥션에서 동시에 동일한 자원을 요청할 경우 순서대로 하나의 커넥션만 번경할 수 있게 해주는 기능이다.

## 락의 기본 동작
- 데이터를 어떻게 다룰 것인지(읽기vs쓰기)에 따라 스레드의 접근을 제어하는 락의 기본 성격이다. 

### 공유 락(Shared Lock, S-Lock)
- 데이터를 조회(Read)할 때 사용한다. 하나의 자원에 대해 여러 세션이 동시에 공유락을 획득할 수 있다. 즉, 여러명이 동시에 읽는 것은 허용된다.
- 공유락이 걸려있는 데이터에는 다른 세션이 데이터를 수정하기 위한 '배타적 락'을 걸 수 없다. 

### 배타적 락(Exclusive Lock, X-Lock)
- 데이터를 수정(Insert, Update, Delete)할 때 사용한다. 
- 특정 세션이 배타적 락을 획득하면 그 락이 해재(Commit/Rollback)될 떄까지 다른 어떤 세션도 해당 데이터에 접근할 수 없고 대기해야 한다. 

## 락의 적용 방식
- 데이터 충돌이 얼마나 자주 일어날지 개발자가 예측하고 접근하는 설계적 관점이다. 

### 비관적 락
- 데이터를 조회하는 시점부터 데이터 수정시 분명히 충돌이 날 것이다라고 가정하고 배타적 락을 걸어버리는 방식이다. 
- 주로 SELECT FOR UPDATE 구문을 사용하며, 하나의 트랜잭션이 끝날 떄까지 다른 트랜잭션들은 대기한다. 
- 특징: 정합성은 완벽히 보장되지만 성능 저하나 데드락의 위험이 있다. 

### 낙관적 락
- 충돌이 안나겠지라고 가정하고 락 없이 데이터를 읽은 후, 데이터를 저장(Commit)하는 시점에 충돌 여부를 확인하는 방식이다.
- 주로 데이터에 version 필드를 두고, 수정할 때 내가 처음 읽은 버전과 일치하는지 조건을 걸어 업데이트한다.(WHERE id=1 AND version=1) 만약 그 사이 버전이 바뀌었다면 예외를 터뜨린다. 
- 특징: DB 자원을 선점하지 않아 동시 처리 성능이 좋지만, 충돌 시 실패 처리를 개발자가 별도로 구현해야한다. 

## 락의 설정 범위
- 동시성 성능을 결정짓는 가장 중요한 요소로, "락을 어느 크기의 단위로 잠글 것인가"에 대한 기준이다. 

### Row Lock(행 락)
- 범위: 테이블 안의 특정 행(Record) 딱 한 줄에만 락을 건다. 
- 특징: 동시성이 가장 높다. 내가 A 데이터를 수정하고 있어도 다른 세션은 B 데이터를 자유롭게 수정할 수 있다. 웹 서비스에서 가장 흔하게 쓰인다. 

### Page/Block Lock
- 범위: 데이터가 저장되는 물리적 페이지(또는 블록)에 락을 건다. 
- 특징: Row Lock보다는 범위가 넓고 Table Lock보다는 좁다. 해당 페이지에 포함된 여러 행이 같이 잠기게 된다. 

### Table Lock
- 범위: 테이블 전체에 락을 건다. 
- 특징: 특정 세션이 테이블을 잠그면 다른 세션의 그 테이블의 어떤 행도 건드릴 수 없다. 주로 테이블 구조를 바꾸는 DDL 작업이나 대용량 배치 작업을 할 떄 사용된다. 

### Database Lock
- 범위: 데이터베이스 인스턴스 전체에 락을 건다. 
- 특징: 락의 범위 중 가장 넓다. 하나의 세션이 이 락을 쥐면 다른 세션의 DB 내의 어떤 테이블도 수정할 수 없다. 전체 DB 백업이나 대규모 마이그레이션 시에만 제한적으로 사용된다.
