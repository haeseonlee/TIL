# TCP vs UDP

## 두 프로토콜의 차이점?
두 프로토콜은 데이터를 목적지에 전달한다는 목적은 같지만, 신뢰성을 우선으로 할 것인지 아니면 속도와 효율성을 우선으로 할 것인가에 따라 동작 방식이 완전히 갈린다.

## TCP(Transmission Control Protocol) 특징
TCP는 **안전하고 정확한 데이터 전달**에 모든 역량을 집중한 프로토콜이다.

1. 연결지향형(Connection-oriented)  
   - 데이터를 주고받기 전에 송신 측과 수신 측이 서로 연결되어 있는지 확인하는 과정을 거친다.
     - 연결 시작: 3-Way Handshake 과정을 통해 논리적인 가상 연결 통로 개설
     - 연결 종료: 데이터 전송이 끝나면 4-Way Handshake를 통해 안전하게 통로 닫음

2. 신뢰성 보장과 제어 기능
   - TCP 헤더에는 순서 번호(Sequence Number)와 확인 응답 번호(Acknowledgment Number)가 포함되어 있어, 흐름 제어(Flow Control)/혼잡 제어(Congestion Control)/오류 제어 및 재전송(Error Control)

3. 데이터 스트림
   - TCP의 경우 데이터의 경계를 구분하지 않고 하나의 연속된 데이터 스트림을 전송한다. 수신측은 애플리케이션에서 필요한 크키만큼 나누어 읽어야 한다.

## UDP(User Datagram Protocol) 특징
UDP는 제어 기능을 과감히 버리고 **빠르고 단순한 데이터 전달**에 집중한 프로토콜이다. 

1. 비연결형(Connectionless)
   - TCP와 달리 사전 연결 설정(Handshake) 과정이 전혀 없다. 수신 측이 데이터를 받을 준비가 되었는지, 살아있는지 확인하지 않고 그냥 목적지 주소를 적어 패킷을 일방적으로 던진다.

2. 신뢰성 없음
   - 데이터가 도중에 유실되거나, 순사가 뒤바뀌어도 UDP는 자체적으로 이를 바로잡지 않는다.
     - 헤더에 Checksum 필드가 있어 패킷이 오염되었는지 확인하지만, 오염된 패킷을 발견하면 수신 측을 이를 그냥 패기하고 재전송을 요청하지 않는다.
  
3. 가벼움과 실시간성
   - 연결을 유지하거나 패킷을 제어할 필요가 없으므로 헤더 크기가 단 8바이트로 크기가 매우 작다. 제어에 드는 리소스가 없기 때문에 순수 전송 속도는 TCP보다 훨씬 빠르고 지연이 최소화된다.

4. 데이터그램 단위
   - 명확한 데이터의 경계를 가진다. 송신 측에서 보낸 하나의 패킷 크기가 수신 측에서도 그대로 유지된다. 

## TCP vs UDP 핵심 차이점 비교
| 구분 | TCP (Transmission Control Protocol) | UDP (User Datagram Protocol)           |
| :--- |:------------------------------------|:---------------------------------------|
| **연결 방식** | **연결 지향형** (Connection-oriented)    | **비연결형** (Connectionless)              |
| **패킷 교환 방식** | 가상 회선 방식 (Virtual Circuit)          | 데이터그램 방식 (Datagram)                    |
| **전송 단위** | 세그먼트 (Segment / 스트림 형태)             | 데이터그램 (Datagram)                       |
| **신뢰성 & 순서** | 높음 (보장됨)                            | 낮음 (보장되지 않음)                           |
| **제어 메커니즘** | 흐름 제어, 혼잡 제어, 오류 제어 제공              | 제어 기능 없음 (최소한의 오류 검출만)                 |
| **헤더 크기** | 20 ~ 60 바이트 (가변적, 무거움)              | 8 바이트 (고정적, 가벼움)                       |
| **속도** | 상대적으로 느림                            | 상대적으로 빠름                               |
| **통신 형태** | 1:1 (Unicast)                       | 1:1, 1:N (Multicast), 1:전체 (Broadcast) |

## TCP를 사용하는 서비스
데이터 유실이 치명적인 경우

- HTTP/HTTPS(웹 브라우징): 웹 페이지의 텍스트나 이미지가 일부 유실되면 페이지가 꺠지기 때문에 신뢰성이 필수다.
- FTP(파일 전송): 파일 다운로드 중 단 1바이트만 깨져도 파일 전체가 손상되므로 완벽한 전송이 필요하다.
- SMTP(이메일 전송): 메일 내용이 유실되거나 순서가 뒤바뀌면 안된다.
- Database 연동 및 SSH 로그인을 위한 통신

## UDP를 사용하는 서비스
지연 시간이 짧아야 하고 약간의 유실은 괜찮은 경우

- 실시간 스트리밍(YouTube 라이브 등): 방송 중 미세하기 프레임이 깨지거나 음성이 튀는 것 보다 화면이 멈추거나 버퍼링이 걸리는 것이 사용자 경험에 더 치명적이다.
- 온라인 게임(FPS, RPG 등): 실시간으로 캐릭터의 좌표를 주고 받아야 한다. 0.1초 전의 유실된 위치 정보보다 현재 위치 정보를 빠르게 받는게 중요하다.
- VoIP(인터넷 전화, 화상 회의): 음성 데이터가 유실되면 잠깐 지지직거릴 뿐이지만, 지연되면 대화가 불가능해진다.
- DNS(도메인 네임 시스템): 단순한 단발성 요청과 응답만 주고받으므로, Handshake 오버헤드가 큰 TCP보다 가벼운 UDP가 더 효율적이다.
